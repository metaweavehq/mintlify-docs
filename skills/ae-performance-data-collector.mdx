## What it does

The **AE Performance Data Collector** gathers a snapshot of a vessel's auxiliary engine performance — running hours, load profile, SFOC, exhaust temperatures, alerts — and files it as evidence under the vessel's case folder. It is the data-gathering counterpart of the [AE Performance Analyzer](/skills/ae-performance-analyzer): the collector fetches, the analyzer interprets.

The collector never makes judgements. It records what the vessel reported, with timestamps, so every later review traces back to the same source of truth.

## When to use it

Trigger this skill any time someone asks for fresh AE numbers, a generator check, or a routine sweep:

- "Get me the latest AE data for POSUN"
- "Collect AE performance for the fleet"
- "AE check on AQUILA"
- "Pull aux engine numbers"

## What it captures

For each auxiliary engine on the vessel:

- Running status (running / standby / fault / maintenance)
- Running hours and hours since last overhaul
- Load percentage and power output
- SFOC (actual vs design)
- Exhaust temperature
- Active maker alerts and severity

Plus vessel-level rollups:

- Number of engines running vs standby
- Total fleet running hours
- Upcoming maintenance windows
- Overall status indicator

## How it works

1. Looks up the vessel by IMO number.
2. Pulls the latest auxiliary-engine telemetry from the connected ERP / demo database.
3. Writes a timestamped evidence record under the vessel's case folder.
4. Updates the case file so the analyzer (and any reviewer) can see the new evidence is available.

If the vessel has no AE data on record, the collector reports that explicitly rather than fabricating values.

## What it produces

Three pieces of evidence are filed for every collection:

- **Raw telemetry** — the structured record consumed by the analyzer
- **Plain-English summary** — a five-line overview a human can read at a glance
- **Filing metadata** — collection timestamp, source, evidence sequence number

Each collection gets its own sequential evidence ID so history is preserved — re-running the collector never overwrites earlier snapshots.

## Why it's separate from the analyzer

Splitting collection from analysis means:

- **Audit trail** — the same raw evidence can be reviewed by multiple agents or humans
- **Reproducibility** — analysis can be re-run without re-fetching, so reviews are stable
- **Cheap to run** — collection is a database query, not a model call
