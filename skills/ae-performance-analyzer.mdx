## What it does

The **AE Performance Analyzer** is a senior agent skill that turns raw auxiliary engine telemetry into a structured technical review. It evaluates engine availability, fuel efficiency (SFOC), power-management posture, overhaul timing, and **blackout risk** — then writes a narrative expert review and decides whether to escalate to a Technical Superintendent.

It runs after a junior data-collector has already gathered the latest evidence for a vessel's auxiliary-engine domain. The analyzer never re-fetches data — it only interprets it, so reviews remain consistent and traceable to source.

## When to use it

Trigger this skill whenever someone asks about auxiliary engine condition, generator status, or blackout exposure — even informally:

- "How are the AEs?"
- "Generator status on POSUN"
- "Any blackout risk on the fleet?"
- "Is AE2 due for overhaul?"
- "Run AE review on AQUILA"

## What it looks at

| Input | Description |
|-------|-------------|
| Engine telemetry | Per-engine running status, hours, load, power, exhaust temperature, SFOC, alerts |
| Collector context | Plain-English summary written by the data-collector that gathered the evidence |
| Case status | Whether this evidence has already been reviewed and any prior decisions |

## How it works

The skill runs a deterministic analysis script that produces structured findings, then writes a narrative expert review based on those findings.

### 1. Engine availability

For each AE the script classifies status — **RUNNING**, **STANDBY**, **FAULT**, or **MAINTENANCE** — and assesses fleet-level redundancy:

| Running | Standby | Assessment | Risk |
|---------|---------|------------|------|
| 1 | 2+ | Adequate redundancy | LOW |
| 1 | 1 | Minimum redundancy | MEDIUM |
| 1 | 0 | No redundancy | HIGH |
| 2+ | 0 | Load sharing, no backup | MEDIUM |

It also tracks utilization patterns — checking whether one engine always carries the load (uneven wear) or whether standby rotation is happening.

### 2. SFOC analysis (fuel efficiency)

Specific Fuel Oil Consumption is load-dependent, so the analyzer never compares raw values across operating points. Instead:

1. Pulls the maker's design SFOC curve at the engine's current load.
2. Calculates variance:

$$
\text{Variance \%} = \frac{\text{SFOC}_\text{actual} - \text{SFOC}_\text{design}}{\text{SFOC}_\text{design}} \times 100
$$

3. Smooths with a 3-point moving average to detect degradation trends.
4. Computes daily fuel-cost impact:

$$
\Delta C_\text{day} = \frac{(\text{SFOC}_\text{actual} - \text{SFOC}_\text{design}) \times P \times 24}{10^{6}} \quad \text{[MT/day]}
$$

$$
\text{Cost}_\text{day} = \Delta C_\text{day} \times \text{Price}_\text{fuel} \quad [\$]
$$

A sustained variance above 8% is flagged. Above 10% triggers escalation.

### 3. Power management &amp; blackout risk

The script evaluates load profile, capacity margin, and load-sharing health to assign a **blackout risk** rating (LOW / MEDIUM / HIGH / CRITICAL). HIGH triggers immediate escalation.

### 4. Overhaul planning

For each engine the analyzer computes hours remaining to next overhaul:

$$
H_\text{remaining} = H_\text{interval} - (H_\text{current} - H_\text{last overhaul})
$$

Tiering:

- $H_\text{remaining} < 0$ — overdue, CRITICAL
- $0 \leq H_\text{remaining} < 500$ — imminent, escalate
- $500 \leq H_\text{remaining} < 2000$ — port-call coordination
- $H_\text{remaining} \geq 2000$ — healthy

### 5. Exhaust temperatures &amp; alerts

Normal range ~350–380 °C. The script flags above 400 °C as WARNING and above 420 °C as CRITICAL, plus any maker alert tagged CRITICAL in the data feed.

## What the review contains

The expert review is a TSI-grade narrative covering:

- **Executive summary** — overall status and headline findings
- **Engine availability** — per-engine status table, availability percentage, redundancy assessment
- **SFOC analysis** — per-engine efficiency, variance vs design, daily cost impact, probable root causes
- **Power management** — load profile, blackout risk, capacity margin
- **Overhaul planning** — urgent and upcoming overhauls, port-call coordination
- **Root cause** — primary cause, confidence level, supporting evidence
- **Recommendations** — prioritized actions with timeline and estimated cost
- **Escalation decision** — whether the case is going to a Technical Superintendent

Every figure in the review traces back to the underlying engine data — nothing is invented.

## Escalation

The analyzer escalates to a Technical Superintendent automatically when any of these conditions are met:

| Trigger | Severity |
|---------|----------|
| Any engine in FAULT status | CRITICAL |
| Blackout risk = HIGH | CRITICAL |
| Overhaul overdue or under 500 hours away | CRITICAL |
| Multiple engines showing WARNING | HIGH |
| SFOC variance above 8% on any engine | HIGH |
| Load-sharing failure | HIGH |
| Exhaust temp above 420 °C | CRITICAL |

Escalation routes the case directly into the TSI's queue with priority, reason, and key findings attached — no manual hand-off needed.

## Why script-driven

Pattern detection, root-cause logic, and cost impact all live in the analysis script — not the language model. This keeps reviews **consistent across vessels**, **auditable** (every number traces to evidence), and **cheap to run** (the model interprets, doesn't recompute).
